### search_actions Usage
The `search_actions` tool allows you to discover the available functionality within this MCP server.
Provide a `query` to search for specific tools or actions. It returns a list of matching tools that can be executed on `.fdx` files, helping you understand how to interact with the script data.

### Cache & file lifecycle
`read_fdx` loads a document into the server's 4-slot LRU cache; most read and edit tools require this to have been called first. `save_fdx` persists the cached document back to disk. `get_cache_status` reports what's currently cached and whether any entry has unsaved edits ("dirty"). `close_fdx` deliberately evicts a document from the cache (refusing if dirty unless `force=true`), and `reload_fdx` force-refreshes a cached document from disk, discarding unsaved edits (also refusing unless `force=true`). `new_file` creates a brand-new blank screenplay and needs no prior `read_fdx` call.

### Scope of editing vs. preservation
The `edit_*` and `new_file` tools operate on paragraphs, element settings, the title page, headers/footers, the six SmartType dictionaries, dual dialogue, and the spell-check ignore words. Every other top-level FinalDraft block — revisions, macros, tag data, actors, cast, page layout, watermarking, and the rest — is preserved via a generic, order-preserving XML tree so it round-trips through `read_fdx` → `save_fdx` losslessly, but there is still **no `edit_*` tool for them**: `get_macro_alias`/`get_macro_alias_list` read the Macros block, `get_revisions`/`get_tag_data`/`get_display_boards` read the Revisions/TagData/DisplayBoards blocks, and the rest are read-only — to modify them, edit the file in FinalDraft. Nested `<DualDialogue>` blocks and the top-level `<SpellCheckIgnoreLists>` block are the two areas with dedicated typed accessors and round-trip losslessly.

### Analysis suite
`get_script_stats`, `get_scene_index`, `get_character_appearances`, `get_page_map`, `get_scene_properties`, and `get_scene_arc_beats` are read-only tools that derive holistic metadata from the loaded document (scene catalog, pagination, character frequency, arc beats) rather than exposing a single FinalDraft block, returning JSON. They share one computation engine (`src/tools/breakdown.ts`), so results are consistent whether queried individually or via the combined `get_fdx_breakdown` report. `get_fdx_breakdown` itself does not return JSON or inline content — it writes a combined text/html/pdf report to a `targetPath` file on disk (see below), reusing the same `breakdown.ts` builders plus its own renderers in `src/tools/breakdown-report.ts` and `src/tools/breakdown-pdf.ts`. None of these tools modify the document or trigger SmartType updates.

### get_fdx_breakdown output
Because a full script breakdown (especially the `pdf` format) can be large, `get_fdx_breakdown` never returns the report inline — it always writes to the file path given in `targetPath`, and the tool result is just a short confirmation message. Callers (including the assisting AI) can subsequently open, read, or otherwise manipulate the written file directly. Choose an extension matching `asType` (`.txt`, `.html`, or `.pdf`); `asType` defaults to `'text'` when omitted.
